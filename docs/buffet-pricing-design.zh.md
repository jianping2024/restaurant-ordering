# 自助餐与基础人头费：产品设计说明（单店版）

## 1. 文档目的与范围

本文描述在**仅支持当前单店**前提下，如何设计「自助餐基础人头费 + 正常点餐/加餐」的完整方案，包括：

- 业务规则与计费口径（含打折）
- 后台配置界面与操作流程
- 与并台、转台、结账的兼容性
- 权限与风控
- 数据模型与实现要点（便于研发落地）

**不在本文展开**：连锁店多门店维度（文末给出扩展预留说明）。

---

## 2. 业务场景与核心原则

### 2.1 两种计费模式（同一套订单体系）

| 模式 | 触发条件 | 应收构成 |
|------|----------|----------|
| **纯点餐** | 订单中**无**自助餐基础费项 | 仅菜品与其它费用 |
| **含自助餐** | 订单中**存在**自助餐基础费项 | **基础人头费** + 菜品（加餐）+ 其它费用 |

原则：**是否收取人头基础费，完全由「是否存在已确认的自助餐基础费明细」决定**，不依赖桌态或隐式标记单独判断，避免状态不一致。

### 2.2 「自助餐」在产品上的定位

- **自助餐 = 可选的基础收费项**（按成人/儿童人数 × 规则价），不是替代菜单计价的全局开关。
- **加餐**仍为普通菜品行，与自助餐基础费**并列计入应收**。
- **统计口径分离**：
  - **加餐金额 / 加菜统计**：仅统计普通菜品行，不把基础人头费算进「加餐」。
  - **应收总额**：基础费 + 加餐 + 其它 − 优惠，全部参与结账。

### 2.3 打折

- 折扣的**计算基数**建议为「可打折小计」：默认包含 **自助餐基础费 + 菜品（加餐）** 中标记为可打折的行。
- 具体某类费用是否参与打折（如服务费），通过行级 `is_discountable` 控制，避免「整单打折」误伤或漏算。

## 3. 权限与风控

### 3.1 为什么限制顾客直接落单

多人就餐场景下，顾客端直接确认自助餐人数易产生**漏人、逃单、纠纷**。建议：

- **顾客端**：可「申请/预选」自助餐（可选），**不产生计费行**或仅产生待确认状态。
- **员工端**：核对人数后**确认**，生成正式的自助餐基础费明细；后端强校验角色，**禁止仅靠前端隐藏入口**。

**当前实现（与代码一致，见 `docs/table-session-close.zh.md` 旁路说明与 `src/lib/guest-table-ordering.ts`）：**

- **顾客扫码点菜**：在服务员完成「自助餐入账」（`POST .../staff/waiter/buffet`）且该餐次存在**未作废的 `buffet_base` 行**之前，**不能**通过 `orders/append` 下单；服务端返回 `buffet_required`，菜单页展示等待提示。
- **服务员代点**（菜单 `waiter_flow`）或 **演示模式** 不受此限制。
- **人数与金额**：成人/儿童可为 **0**（含 0 人、0 € 入账仅用于开台激活）；须仍有有效价格规则（单价可为 0）。

### 3.2 建议角色能力

| 动作 | 建议角色 |
|------|----------|
| 配置时段、价格规则、节假日日历 | 店长 / 管理员 |
| 确认自助餐人数、追加/减少人数（需审计） | 服务员（减人可提高权限门槛） |
| 改价、作废基础费、冲正 | 收银 / 店长 |
| 查看报表 | 按岗位分配 |

### 3.3 审计

对自助餐基础费建议记录：**创建人、修改人、修改前后人数与金额、时间、原因**（与现有订单审计策略对齐）。

## 4. 定价与配置设计（简单易操作）

单店场景下，配置分为四块：**自助餐定义、时段、价格规则、日历**。后台菜单可按「自助餐设置」一个入口，内嵌 Tab 或左侧导航。

### 4.1 自助餐定义（Buffet Product）

**页面目标**：说明「卖的是哪种自助」，不在这里堆复杂价格。

- **名称**（如：午市自助、海鲜自助）
- **启用 / 停用**
- **顾客可见策略**：可选「顾客可见需员工确认」或「仅员工可选」
- **展示说明**（可选）：用餐时长、浪费规则等（顾客端展示用）

### 4.2 时段（Time Slots）

**页面目标**：把「午市 / 晚市」等抽象成可复用对象，供规则引用。

每条时段建议字段：

- **名称**（午市、晚市、夜宵）
- **每日起止时间**（支持跨日，如 22:00–02:00）
- **适用星期**（周一至周日多选；单店可按实际勾选）
- **排序**（列表展示顺序）

### 4.3 价格规则（核心列表页）

**页面目标**：运营人员最常操作；列表要一眼看出「当前生效价」。

**列表筛选**：自助餐、时段、日历类型（平日/周末/节假日/特殊）、生效状态（生效中/未开始/已过期）、启用状态。

**列表列建议**：自助餐名称、时段、日历类型、生效日期范围、成人价、儿童价、优先级、状态、操作（编辑 / 复制 / 停用）。

**新建/编辑表单字段**：

| 字段 | 说明 |
|------|------|
| 自助餐 | 必选 |
| 时段 | 必选 |
| 日历类型 | 必选：平日 / 周末 / 节假日 / 特殊 |
| 生效日期范围 | `valid_from` ~ `valid_to` |
| 成人价、儿童价 | 必填，单位与系统一致 |
| 优先级 | 数字越大越优先；用于同日历维度下多条规则时 |
| 备注 | 如「五一活动价」 |
| 启用 | 开关 |

**必做交互（降低配错率）**：

1. **保存前预览**：根据「当前时间 + 未来 7 天」展示将命中的规则与价格（读模型即可）。
2. **冲突提示**：同一自助餐 + 同一时段 + 重叠日期范围 + 同一日历类型下多条启用规则时，提示「将按最高优先级生效」，并列出候选规则。
3. **一键复制**：从「平日午市」复制为「节假日午市」，只改日历类型与价格。

### 4.4 日历（节假日 / 特殊日）

**页面目标**：把「是不是节假日」从代码里拿出来，避免每年改代码。

- **月历视图**：点选日期设置标签：`工作日默认` / `周末`（若你们不用系统周末可隐藏）/ `节假日` / `特殊活动日` 等。
- **批量设置**：按起止日期区间批量标记（如国庆 7 天）。
- **导入**（可选）：CSV 批量导入，适合连锁前演练。

**匹配建议**：某日先查日历表；若无显式记录，再回退为「是否周末」等默认策略（单店可简化为：只维护「节假日/特殊」覆盖平日即可）。

## 5. 下单与计价流程（与现有会话/订单对齐）

以下与现有 `table_sessions`、`orders`、并台方案兼容：自助餐与普通菜品均为**订单行**，并台时迁移的是**订单及行项目**，不依赖「桌子的隐藏模式」。

### 5.1 推荐流程

1. 开台 / 会话开始（现有能力）。**实现契约**（单管道、no-op、一次返回 detail）见 [`buffet-open-table.zh.md`](buffet-open-table.zh.md)。
2. **纯点餐**：员工或顾客点菜品 → 仅产生 `menu_item` 类行。
3. **选自助餐**：
   - 顾客可发起申请（可选）；
   - **员工确认**成人人数、儿童人数（及可选年龄段规则）；
   - 系统按「门店当前时间 + 日历 + 时段」匹配规则，生成 **自助餐基础费行**（建议 `item_type = buffet_base_fee`），**单价与行金额落单锁定**。
4. **加餐**：继续点菜品 → 仅增加 `menu_item` 行；**不改变**已锁定的基础费（除非走「改人数/改价」受控流程）。
5. 结账：合计 = 基础费小计 + 菜品小计 + 其它 − 折扣。

### 5.2 价格锁定

规则表变更**不影响已开台订单**。已生成的自助餐行应保存：

- 命中的规则 ID（可选，便于对账）
- 行内成人单价、儿童单价或合并行金额
- 人数快照

如需随规则重算，必须走**显式操作**（权限 + 审计），避免静默改价引发纠纷。

## 6. 金额口径与打折（实现约定）

建议在订单聚合层（或物化字段）明确区分：

| 口径 | 包含内容 | 典型用途 |
|------|----------|----------|
| `food_subtotal` | 仅 `menu_item` | 加餐统计、顾客端「加菜金额」 |
| `buffet_base_subtotal` | 仅 `buffet_base_fee` | 基础人头费小计、报表 |
| `discountable_subtotal` | `is_discountable = true` 的菜品 + 基础费（可配置） | 折扣、满减基数 |
| `order_total` | 全量合法计费行 + 费用 − 优惠 | 应收 |

打折分摊（建议）：折扣总额按 `discountable_subtotal` 内各行金额比例分摊到行级 `discount_allocated`，便于退菜、退款与对账。

## 7. 并台、转台与自助餐

**并台（产品定稿）：** 见 [并台产品规则](./table-merge-product.zh.md) §5（同类型合并人数、并台时刻 `resolve_buffet_prices`、一条 `buffet_base`、多类型拒绝并台）。

以下表格描述**旧并台实现**（订单迁目标会话、行原样归属），供对照；新实现以 `table-merge-product.zh.md` 为准。

| 场景 | 行为 |
|------|------|
| 两边都无自助餐 | 仅菜品合并，逻辑不变 |
| 一边有、一边无 | 合并后若存在 `buffet_base_fee` 行，则应收含基础费；无自助餐一侧原菜品仍计费 |
| 两边都有自助餐 | 合并后出现多条基础费行或需在 UI 上提示「是否合并人数」；**推荐产品策略**：并台前提示员工确认**合并后的成人/儿童总数**，合并为一条基础费行（或保留多条但展示总人数，二选一需在实现层统一） |

**原则**：不在并台逻辑里「重新推断是否自助餐」；以**订单行是否存在基础费**为准。若需合并人数，应通过受控 RPC 更新行并记审计。

## 8. 数据模型建议（单店，可预留扩展）

以下为逻辑表，命名可按现有库规范调整。

### 8.1 `buffets`（自助餐定义）

- `id`, `name`, `is_active`, `customer_visible`, `requires_staff_confirm`, `description`, `created_at`, `updated_at`

### 8.2 `time_slots`（时段）

- `id`, `name`, `start_time`, `end_time`, `weekdays`（bitmask 或 JSON）, `sort_order`

### 8.3 `buffet_price_rules`（价格规则）

- `id`, `buffet_id`, `time_slot_id`
- `calendar_kind`：`weekday` | `weekend` | `holiday` | `special`
- `valid_from`, `valid_to`
- `adult_price`, `child_price`
- `priority`, `is_active`, `note`

### 8.4 `calendar_overrides`（日历覆盖，单店）

- `date`（date, PK）, `tag`（`holiday` / `special` / …）

### 8.5 订单行扩展

在 `order_items`（或等价表）建议：

- `item_type`：`menu_item` | `buffet_base_fee` | 其他费用类
- `buffet_id`（可空）, `adult_count`, `child_count`（或两行拆分成人/儿童，二选一）
- `unit_price`, `line_total`, `is_discountable`
- `price_rule_id`（可空）, `locked_at`

### 8.6 规则匹配函数（伪逻辑）

输入：`now`, `buffet_id`  
步骤：

1. 取 `date = now::date`，查 `calendar_overrides` 得 `tag`；若无则根据星期得到 `weekday/weekend`。
2. 根据 `now::time` 与 `weekdays` 命中 `time_slot`（跨日时段需单独处理）。
3. 在 `buffet_price_rules` 中筛选：`buffet_id`、命中 `time_slot`、`calendar_kind` 匹配或兼容、`valid_from/valid_to` 包含今日、`is_active`。
4. 取 `priority` 最大一条；同优先级时定义二级排序（如 `valid_from` 新者优先）。

## 9. 报表与对账（简要）

- **收入结构**：基础费收入 vs 菜品收入（按 `item_type` 汇总）。
- **自助餐 KPI**：桌数、人数（成人/儿童）、时段分布、节假日 vs 平日占比。
- **异常**：有菜品但长期无基础费、并台后人数与基础费不一致提示（可选，规则引擎）。

## 11. 验收要点（摘要）

- [ ] 无自助餐时，订单仅菜品计费，应收正确。
- [ ] 有自助餐时，基础费 + 加餐同时计入应收；加餐统计不含基础费。
- [ ] 折扣基数包含基础费与可打折菜品时，总额与分摊正确。
- [ ] 仅员工（或配置的角色）可生成/修改自助餐基础费行；顾客端无法绕过。
- [ ] 规则变更不影响已锁定订单；并台迁移订单行后结账正确。
- [ ] 后台配置：时段、规则、日历三类页面可独立完成日常调价，且具备预览与冲突提示。
