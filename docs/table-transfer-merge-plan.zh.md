# 转台与并台实施方案

## 范围

本文定义转台与并台功能的简化实现方案，目标是：

- 以“当前状态正确”为第一优先
- 并台后自动形成单账单流
- 最小化数据结构变更
- 通过后端原子操作避免半成功状态

## 业务规则

### 转台

- 将来源桌的一个活跃餐次迁移到目标桌。
- 目标桌不能已有活跃餐次。
- 会话、订单、活跃结账请求需一并更新。

### 并台

- 将来源桌活跃餐次并入目标桌活跃餐次。
- 来源桌与目标桌都必须有活跃餐次。
- 来源订单改挂到目标会话，并更新为目标桌号。
- 来源会话关闭并记录并台关系。
- 结账流收敛为目标会话下的一条活跃账单流。

## 数据模型变更

### 复用现有表

- `table_sessions`
- `orders`
- `bill_splits`

### 新增字段

- `table_sessions.merge_into_session_id`（可空 UUID）
- `table_sessions.closed_reason`（可空文本）

### `closed_reason` 常见取值

- `merged`：并台时关闭**来源桌**会话。
- `waiter_closed`：服务员在观察页对当前桌点击 **「关台」** 结束餐次。
- 老板在后台确认收款关台时，会话标记为 `closed`（实现上可不依赖 `closed_reason` 区分）。

## 后端操作设计

采用两个 RPC 作为原子写入边界：

- `transfer_table_session(p_restaurant_id, p_from_table, p_to_table)`
- `merge_table_sessions(p_restaurant_id, p_source_table, p_target_table)`

### 转台流程

1. 锁定来源桌活跃会话。
2. 校验目标桌活跃会话冲突。
3. 更新来源会话桌号。
4. 更新关联订单桌号。
5. 更新活跃 `bill_splits` 桌号。

### 并台流程

1. 锁定来源桌与目标桌活跃会话。
2. 将来源订单迁移到目标会话/桌号。
3. 合并活跃结账记录，收敛到目标账单流。
4. 关闭来源会话并设置：
   - `status = closed`
   - `closed_reason = merged`
   - `merge_into_session_id = 目标会话ID`

## 前端实施计划

主要入口：

- `src/components/dashboard/TablesManager.tsx`

交互设计：

- 活跃餐次列表
- `转台` 操作按钮
- `并台` 操作按钮
- 确认弹窗（来源桌、目标桌选择）
- 明确展示并台不可逆语义与风险提示

## 一致性策略

- 通过 RPC 避免前端串行多次写入造成中间态。
- 对会话状态变化采用乐观失败提示：
  - “桌台状态已变化，请刷新后重试。”
- 操作成功后立即刷新活跃餐次数据。

## 服务员关台与看板可见性

### 后厨 / 服务员看板上的订单范围

- 两页在通过密码进入后，会订阅 Supabase Realtime（`orders`、`table_sessions`），并在进入时立即拉取最新数据。
- **仅展示仍挂在活跃餐次上的订单**：`table_sessions.status` 为 `open` 或 `billing` 时，对应 `orders.session_id` 的订单会出现在看板；`session_id` 为空时视为兼容旧数据，仍会显示。
- 餐次一旦关闭（结账、并台来源桌、服务员关台等），这些订单不再满足「活跃会话」条件，**从后厨与服务员看板消失**，无需整页手动刷新（依赖订阅回调或再次进页拉数）。

### 服务员「关台」

- 入口：`/[slug]/waiter` 桌卡上的「关台」（与「转台」「并台」并列）。
- **显示条件**：该桌汇总上 **没有「制作中」、没有「可端菜」**（厨师尚未把任何菜做到可出餐状态）。在此条件下，无论仅有待做、仅有已取消、或待做与已取消混合，均可关台。
- **不显示「关台」**：存在制作中或可端菜时，需先完成后厨出餐流程或由服务员逐单处理，避免与出餐状态冲突。
- **效果**：将本桌当前活跃 `table_sessions` 更新为 `closed`，写入 `closed_at` 与 `closed_reason = waiter_closed`；不迁移订单行，仅结束会话。看板按上文规则自动隐藏该桌相关订单。

## 验证清单

功能验证：

- 转台后会话/订单/结账请求桌号一致更新。
- 并台后仅保留一个活跃会话与一条活跃账单流。
- 来源会话被正确关闭并关联到目标会话。

并发验证：

- 同一来源桌的并发操作不会产生重复活跃状态。
- 第二个冲突操作能友好失败并提示刷新。

跨页面验证：

- 顾客端菜单/账单、后厨看板、服务员看板均正确反映操作后状态。
- 服务员「关台」后，该桌订单从服务员与后厨看板消失，顾客端新开餐次可正常点单。

## 上线说明

- 启用 UI 前先执行数据库迁移。
- 除桌台管理操作面板外，不改动既有主要业务流程。
- 后续如有需要可加 feature flag 做灰度控制。
