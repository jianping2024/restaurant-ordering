# 转台与并台实施方案

## 范围

本文定义转台与并台功能的简化实现方案，目标是：

- 以“当前状态正确”为第一优先
- 并台后自动形成单账单流
- 最小化数据结构变更
- 通过后端原子操作避免半成功状态

## 业务规则

### 转台

- 将来源桌的一个活跃餐次迁移到目标桌。
- 目标桌不能已有活跃餐次。
- 会话、订单、活跃结账请求需一并更新。

### 并台

- 将来源桌活跃餐次并入目标桌活跃餐次。
- 来源桌与目标桌都必须有活跃餐次。
- 来源订单改挂到目标会话，并更新为目标桌号。
- 来源会话关闭并记录并台关系。
- 结账流收敛为目标会话下的一条活跃账单流。

## 数据模型变更

### 复用现有表

- `table_sessions`
- `orders`
- `bill_splits`

### 新增字段

- `table_sessions.merge_into_session_id`（可空 UUID）
- `table_sessions.closed_reason`（可空文本）

## 后端操作设计

采用两个 RPC 作为原子写入边界：

- `transfer_table_session(p_restaurant_id, p_from_table, p_to_table)`
- `merge_table_sessions(p_restaurant_id, p_source_table, p_target_table)`

### 转台流程

1. 锁定来源桌活跃会话。
2. 校验目标桌活跃会话冲突。
3. 更新来源会话桌号。
4. 更新关联订单桌号。
5. 更新活跃 `bill_splits` 桌号。

### 并台流程

1. 锁定来源桌与目标桌活跃会话。
2. 将来源订单迁移到目标会话/桌号。
3. 合并活跃结账记录，收敛到目标账单流。
4. 关闭来源会话并设置：
   - `status = closed`
   - `closed_reason = merged`
   - `merge_into_session_id = 目标会话ID`

## 前端实施计划

主要入口：

- `src/components/dashboard/TablesManager.tsx`

交互设计：

- 活跃餐次列表
- `转台` 操作按钮
- `并台` 操作按钮
- 确认弹窗（来源桌、目标桌选择）
- 明确展示并台不可逆语义与风险提示

## 一致性策略

- 通过 RPC 避免前端串行多次写入造成中间态。
- 对会话状态变化采用乐观失败提示：
  - “桌台状态已变化，请刷新后重试。”
- 操作成功后立即刷新活跃餐次数据。

## 验证清单

功能验证：

- 转台后会话/订单/结账请求桌号一致更新。
- 并台后仅保留一个活跃会话与一条活跃账单流。
- 来源会话被正确关闭并关联到目标会话。

并发验证：

- 同一来源桌的并发操作不会产生重复活跃状态。
- 第二个冲突操作能友好失败并提示刷新。

跨页面验证：

- 顾客端菜单/账单、后厨看板、服务员看板均正确反映操作后状态。

## 上线说明

- 启用 UI 前先执行数据库迁移。
- 除桌台管理操作面板外，不改动既有主要业务流程。
- 后续如有需要可加 feature flag 做灰度控制。

## 后续可选增强

- 并台前预览（菜品数量、金额变化）
- 最近一次操作短时撤销
- 结构化操作审计日志表
